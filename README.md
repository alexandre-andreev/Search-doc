# README — Как работать с Claude Code по проекту search-doc

Этот документ — для **вас как заказчика**, не для Claude Code. Здесь пошагово описано, как запустить работу.

---

## Что у нас есть в результате pre-MVP

Перед началом работы с Claude Code зафиксируем итоги исследовательской фазы:

**Подтверждено:**
- Семантический поиск по саммари каталога работает на хорошем уровне (MRR=0.76).
- Модель `multilingual-e5-small` справляется с задачей на вашей GTX 1650 Ti (~384 чанков/сек).
- Eval-набор `eval_queries_v2.yaml` содержит 35 реалистичных запросов с проверенными ожиданиями.
- 3 настоящих провала (`pytest`, `golang`, `llm_abbreviation`) решаются добавлением FTS5 поверх семантики.

**Принято как архитектурное решение:**
- **MVP не включает body-extractor** (извлечение текста из PDF/EPUB). Работаем только с метаданными и саммари из `catalog.xlsx`.
- **Используется Вариант B**: основной поиск + блок «related_books» (что может быть полезно по теме).
- Инкрементальный переиндекс — для еженедельного пополнения каталога.

## Как стартовать сессию с Claude Code

Откройте PowerShell в папке проекта и активируйте окружение:

```powershell
cd D:\_project\Search-doc
.\venv-embed-test\Scripts\Activate.ps1
```

Запустите Claude Code в этой папке. Он автоматически увидит файлы проекта.

### Первое сообщение Claude Code

Скопируйте дословно (можно подправить под себя):

> Привет. Я работаю над проектом семантического поиска по локальной библиотеке книг.
> 
> Перед началом реализации:
> 1. Прочитай файл `TASK_FOR_CODE.md` — это полная постановка задачи.
> 2. Прочитай `project_spec.yaml` — это машиночитаемая спецификация.
> 3. Просмотри `pre_mvp_eval.py` — это работающий референсный код, многое будем переиспользовать.
> 4. Просмотри `eval/eval_queries_v2.yaml` — это наш eval-набор.
> 
> После этого скажи: понял задачу, готов начать **Этап 1** (раздел «3. Этапы реализации» в TASK_FOR_CODE.md). Не приступай к коду, пока я не подтвержу.

**Ключевой момент:** не давайте Claude Code «делать всё сразу». Каждый этап — отдельный сеанс. После каждого этапа — проверка acceptance criteria и прогон тестов.

## Последовательность сессий

### Сессия 1 — Этап 1: Каркас + БД + импорт каталога

После того как Claude Code прочитает документы и подтвердит готовность:

> Начинай **Этап 1** из TASK_FOR_CODE.md. Доустанови недостающие зависимости (sqlite-vec, click, tqdm, simhash, pytest) в существующее окружение. Создай структуру проекта, схему БД, модули `catalog_import`, `embedder`, `index`. Реализуй CLI-команды `import` и `status`. Напиши unit-тесты для `safe_int`, `load_catalog`.

**Критерий завершения сессии:**
```powershell
python search-doc.py import
# Должно вывести: "Импортировано 3180 книг, создано ~6300 чанков, время ~25 сек"

python search-doc.py status
# Должно показать статистику БД
```

Если эти команды работают — Этап 1 закрыт. Закрываете чат с Claude Code.

### Сессия 2 — Этап 2: Базовый семантический поиск

Новая сессия. Первое сообщение:

> Этап 1 завершён, БД создана и заполнена. Начинай **Этап 2** из TASK_FOR_CODE.md. Реализуй модули `search/semantic.py`, `search/ranker.py`, CLI-команду `search`, перенеси и адаптируй `pre_mvp_eval.py` в `eval/run_eval.py`. Прогоняй eval после реализации.

**Критерий завершения:**
- `python search-doc.py search "модульное тестирование Python" --format json` возвращает валидный JSON.
- `python eval/run_eval.py` даёт R@5 ≥ 0.78, MRR ≥ 0.74.

### Сессии 3-7 — Этапы 3-7

Аналогично: один этап = одна сессия. После каждой:
- Запускаете eval (после Этапа 3 — должны улучшиться pytest/golang/llm).
- Проверяете acceptance criteria из TASK_FOR_CODE.md раздел 8.
- Если всё ок — следующая сессия.

## Что делать, если Claude Code сбивается с курса

**Если предлагает реализовать body-extractor:**
> Body-extractor вне MVP, см. раздел 9 TASK_FOR_CODE.md. Не отвлекайся от плана.

**Если предлагает поменять модель или embedding-параметры:**
> Параметры зафиксированы в project_spec.yaml. Не менять без согласования.

**Если делает много этапов сразу:**
> Работай по одному этапу. Это требование из раздела 11 TASK_FOR_CODE.md.

**Если хочет пропустить тесты:**
> Тесты не пропускаются. Acceptance criteria для каждого этапа — обязательны.

**Если предлагает «упростить» JSON-схему:**
> JSON-схема — контракт с Codex. Менять только при bump'е schema_version, и только с моего разрешения.

## Что вы делаете между сессиями

После каждого этапа:

1. **Запустите acceptance-команды** из TASK_FOR_CODE.md.
2. **Прогоните eval** (после Этапа 2 и далее):
   ```powershell
   python eval/run_eval.py
   ```
3. **Запустите unit-тесты:**
   ```powershell
   pytest tests/
   ```
4. **Сохраните вывод eval** — это ваша история качества. Если на каком-то этапе метрики деградируют — это сигнал откатиться.
5. **Закройте чат с Claude Code** — следующий этап в новой сессии с чистым контекстом.

## Финальная приёмка MVP

После Этапа 7 — финальная проверка:

```powershell
# Холодный старт с нуля
del cache\semantic_index.sqlite
python search-doc.py import
# Время ≤ 60 сек

# Поиск с related
python search-doc.py search "как стать лучше как программист" --format json
# Должны быть results + related_books

# Прогон eval
python eval/run_eval.py
# R@5 ≥ 0.85, R@10 ≥ 0.80, MRR ≥ 0.80

# Инкрементальный
python search-doc.py import
# Время ≤ 5 сек (нет изменений в catalog.xlsx)

# Тесты
pytest tests/
# Все должны проходить
```

Если все 4 проверки прошли — MVP готов. Подключайте Codex к CLI и пользуйтесь.

## Интеграция с Codex (после MVP)

В Codex настройте custom command/tool, который вызывает:

```
python D:\_project\Search-doc\search-doc.py search "{user_query}" --format json
```

И передаёт результат как контекст для модели OpenAI. Точная настройка зависит от вашего Codex GUI клиента — посмотрите его документацию по custom tools / shell commands.

Пример системного промпта для Codex:

> Когда пользователь задаёт вопрос, который касается книг, программирования, технических тем — вызови инструмент `search-doc`. Из его JSON-ответа покажи пользователю:
> 1. Top-5 книг из `results` с кратким описанием почему они подходят (используй поля title, author, summary, matched_in).
> 2. Если есть `related_books` — отдельным блоком «Вам также может быть интересно по теме».
> 3. Если пользователь спрашивает «открой N-ю книгу» или подобное — вызови `search-doc.py open <book_id>`.

## Что делать после MVP (когда поработаете 1-2 недели)

По результатам реального использования вы поймёте, нужны ли улучшения. Возможные следующие итерации:

- **Body-extractor для PDF/EPUB** — если заметите, что для специфических запросов не хватает контекста.
- **Cross-encoder reranking** — если хотите ещё буст качества (+200MB модель, +1 сек на запрос).
- **OCR для сканов** — если в библиотеке много DjVu/сканов без текстового слоя.
- **GUI** — если устанете печатать в Codex и захотите кнопочный интерфейс.

Ни одно из этих улучшений **не требуется** для работающего MVP. Сначала живите с тем, что получится после Этапа 7.
